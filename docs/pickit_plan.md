# Verdict 기획 문서 v1.0

> 앱인토스 기반 소셜 투표 앱 — "정답 없는 질문에 대한 대중의 평균값"

---

## 1. 서비스 개요

| 항목 | 내용 |
|------|------|
| 서비스명 | Verdict (판정단) |
| 플랫폼 | 앱인토스 (Toss in-app) |
| 핵심 가치 | 살다보면 누가 맞는지 모를 때, 대중의 평균적인 의견을 확인할 수 있는 가벼운 투표 앱 |
| 타겟 유저 | 토스 앱 활성 사용자 (20~40대 주요 타겟) |
| 수익 모델 | 광고 (네이티브 배너 + 리워드 + 일부 전면) + 토스포인트 리워드 직지급 (§7, §13 참고) |

---

## 2. 핵심 컨셉

- 누구나 일상 속 궁금증을 질문으로 등록
- 투표 후 즉시 결과 공개 (투표 전엔 참여자 수만 노출)
- 성별 / 연령대별 통계로 차별화된 결과 제공
- 결과 이미지 링크 공유로 바이럴 유도
- 토스 실명 인증 기반 → 어뷰징 방지 + 결과 신뢰도 확보

## 3. 카테고리 구성

| 카테고리 | 설명 | 예시 질문 |
|----------|------|-----------|
| 일상 | 생활 전반, 습관, 소비 | "카톡 읽씹, 화가 나?" |
| 연애/관계 | 썸, 연애, 이별, 친구관계 | "첫 만남 더치페이 괜찮아?" |
| 직장/학교 | 야근, 직장 눈치, 학교생활 | "야근 수당 없는 야근 그냥 해?" |
| 게임 | 게임 내 판단, 유저 논쟁 | "캐리 못하면 트롤인가?" |
| 기타 | 위 카테고리 외 모든 주제 | - |

> 추후 데이터 기반으로 "스포츠", "육아", "취미" 등 추가 가능

## 4. 화면 구성 (IA)

### 4-1. 홈 피드

- 최상단: 오늘의 투표 카드 (일상 / 게임 / 연애·관계 / 직장·학교 탭 선택 시에만 노출)
- 전체 / 기타 탭에서는 오늘의 투표 카드 미노출, 일반 피드만 표시
- 중단: 유저 투표 피드 (인기순 / 최신순 / 마감임박순 탭 전환)
- 피드 중간: 배너 광고 삽입
- **하단: "지난 오늘의 투표" 섹션** — 날짜별 카드 나열, 탭 시 바로 결과 확인 가능 (영구 보관, §12 참고)
- 각 카드 표시 정보: 카테고리 태그 / 질문 / 참여자 수 / 남은 시간 / 결과 바 (투표 후)

#### 카드 인터랙션 규칙

- **카드 영역 전체가 탭 영역** — 별도의 "자세히 보기" 버튼 없음, 카드 탭 시 상세 페이지로 이동
- **오늘의 투표 카드**: 탭 시 상세 페이지로 이동, 투표는 상세 페이지에서만 가능
- **일반 투표 카드**: 카드 내 선택지 버튼으로 **피드에서 즉시 투표 가능** (간단 투표 UX)
  - 카드 영역(질문 텍스트/카테고리 태그 등) 탭 시 상세 페이지로 이동
  - 외부 URL을 통한 진입은 항상 상세 페이지로 라우팅
  - 피드에서 투표한 경우 카드가 결과 바 형태로 즉시 갱신, 추가 통계는 상세 페이지에서 확인
- **지난 오늘의 투표 카드**: 탭 시 상세 페이지로 이동, 결과만 확인 (재투표 불가)

### 4-2. 투표 상세 / 결과

- 질문 표시
- 투표 전: 선택지 버튼만 노출, 참여자 수만 표시
- 투표 후: 즉시 결과 공개 — **오늘의 투표 / 일반 투표 모두 동일한 풍부한 통계 제공**
  - 전체 결과 (% 바)
  - 성별 비교
  - 연령대별 비교 (20대 / 30대 / 40대+)
- 결과 화면 하단: 배너 광고 (결과 로딩 0.8초 연출 후 노출)
- 링크 공유 버튼 (카카오톡 / 인스타그램 / URL 복사)

#### 투표 진입 동선

| 투표 타입 | 피드에서 투표 | 상세 페이지에서 투표 | 비고 |
|-----------|---------------|---------------------|------|
| 오늘의 투표 | ❌ | ✅ | 카드 탭 → 상세 페이지에서만 투표 |
| 일반 투표 | ✅ | ✅ | 피드 카드 내 간단 투표 + 상세 페이지 투표 모두 지원, URL 진입 시 상세 페이지로 라우팅 |

#### 마감된 투표 진입 (미참여자) — 광고 시청 후 결과 공개

```
마감된 투표 진입 (미참여자)
    ↓
"투표가 종료됐어요"
"결과를 보려면 광고를 시청해주세요"
    ↓
광고 시청 (앱인토스 전면 광고 SDK)
    ↓
결과 공개 ✅
```

- 진입 시 결과 블라인드 + 안내 카드 노출 ("투표가 종료됐어요" / "결과를 보려면 광고를 시청해주세요")
- 주 액션 버튼: "광고 보고 결과 확인하기" (CTA 1개)
- 보조 액션: "홈으로 돌아가기"
- 광고는 **앱인토스 전면(interstitial) 광고 SDK**로 노출, 시청 완료 콜백 수신 후 결과 공개
- 광고 로드 실패 / 사용자 닫음 처리 시 토스트 안내 + 재시도 가능
- 시청 완료 후에는 일반 결과 화면(전체 % / 성별 / 연령대) 동일 노출
- 마감된 투표를 **이미 참여한 사용자**는 광고 없이 결과 즉시 노출 (기존 흐름)

### 4-3. 질문 등록

**일반 투표 트랙**
- 질문 입력 (최대 60자)
- 선택지 입력 (2~5개)
- 카테고리 선택
- 투표 기간 선택: 10분 / 30분 / 1시간 / 6시간 / 24시간
- 등록 시 LLM 자동 검열 통과 후 피드 노출 (§10 참고)

**오늘의 투표 후보 신청 트랙**
- 위 항목 동일
- 추가: "오늘의 투표 신청" 토글 선택
- **1인 1일 1건만 후보 신청 가능** — 후보 풀에서 1개를 선정하는 구조라 다수 등록은 무의미, 질 우선
- **신청 즉시 공개되지 않음** — 신청 당일에는 후보 풀에만 적재되고, 심사·선정 후 **다음날 오전 8시에 발행**돼 카테고리 상단 고정 노출 (§5 선정 흐름 참고)
- 작성 시 5P 즉시 지급, 선정 시 +30P 추가 (§7-1 오늘의 투표 트랙 참고)
- 화면 카피는 "후보 신청"으로 통일 (즉시 공개로 오인 방지) — 제출 버튼 "오늘의 투표 후보 신청하기", 성공 토스트 "후보로 신청됐어요. 선정되면 내일 오전 8시에 공개돼요"

### 4-4. 마이페이지

- 프로필: 익명 닉네임 (`판정단 #A12F` 형식) / 토스 인증 완료 배지
  - 닉네임 hex 부분은 `user.id` 해시로 클라이언트에서 deterministic 생성 (DB 미저장)
  - 항상 동일한 값 노출, 충돌·고유성 관리 불필요
- 통계: 내가 올린 투표 수 / 참여한 투표 수 / 상단 선정 수
- **인구통계 공개 설정**: 성별 / 연령대 (20대 / 30대 / 40대+ / 비공개)
  - 토스 **연계정보 선택동의** 시 자동 반영 (생년월일은 백엔드에서 연령대로 변환 후 저장)
  - 미동의 시 `비공개`로 저장 → 결과 집계 시 별도 '비공개' 카운트로 노출
  - 매 세션 시작 시 토스에서 최신 정보 가져와 자동 동기화 (Edge Function)
  - 마이페이지에서는 **공개/비공개 토글만 가능** — 값 자체는 토스 신원 데이터 기반이라 임의 변경 불가
  - 미동의자 동의 유도 UX: 결과 화면에 "동의하면 연령대별 비율도 볼 수 있어요" 작은 nudge
- 내가 올린 투표 목록 (진행중 / 마감)
- 참여한 투표 목록 + **내 선택 vs 다수의견 일치/불일치 뱃지** (결과 바 위에 표시)

#### 카드 표시 / 인터랙션

- **홈피드와 동일한 카드 컴포넌트** 사용 — 카테고리 태그 / 질문 / 참여자 수 / 결과 바를 인라인으로 표시
- 카드 탭 → 상세 페이지 이동, 성별·연령대별 풍부한 통계는 상세 페이지에서 확인 (§4-2 참고)
- 일반 투표는 최근 **30일치 이력**만 노출, 30일 경과 시 마이페이지에서도 자동 제거 (§12 보관 정책 참고)
- 오늘의 투표 참여 이력은 영구 노출

> 누적 포인트 섹션은 표시하지 않는다. 활동 보상은 토스 포인트로 즉시 지급되며, 잔액/내역은 토스 앱에서 확인한다 (§7 참고).

## 5. 오늘의 투표 선정 시스템

### 선정 흐름

```
유저가 후보 신청
    ↓
LLM 1차 검열 (혐오/중복/품질, §10 — OpenAI gpt-4o-mini)
    ↓
점수 산정 (아래 가중치 합산)
    ↓
상위 3개 후보 → 운영자 최종 확인
    ↓
카테고리별 1개 선정 → 오전 8시 발행
```

### 점수 산정 기준

| 항목 | 가중치 | 비고 |
|------|--------|------|
| 질문 AI 흥미도 점수 | 40% | LLM 평가 (§10) |
| 커뮤니티 추천수 | 25% | 후보 목록 공감 수 |
| 질문자 활동 점수 | 20% | 투표 참여율 + 등록 이력 |
| 계정 신뢰도 | 10% | 가입 기간 + 신고 이력 없음 |
| 카테고리 다양성 | 5% | 전날과 같은 카테고리면 감점 |

> 질문 점수와 질문자 점수를 분리 산정 → 신규 유저도 좋은 질문이면 선정 기회 부여

## 6. 어뷰징 방지 정책

### 1단계 — 토스 인증 기반 (자동)

- 1인 1투표 강제 (토스 실명 계정 기준)
- **일반 투표 등록 일일 캡: 10건** (1~2건 무료, 3~10건 광고 시청 필요, 11건째부터 차단)
- 오늘의 투표 후보 신청은 **1인 1일 1건** 하드캡 (§4-3 참고)

### 2단계 — LLM 자동 검열 (질문 등록 시 — §10 참고)

- 혐오 / 비하 / 정치적 편향 감지 → 즉시 차단
- 중복 질문 유사도 검사 (최근 30일 기준)
- 질문 품질 점수 기준 미달 시 등록 반려 + 사유 안내

### 3단계 — 커뮤니티 신고 + 운영자 검토

- 유저 신고 기능 (부적절한 질문 / 선택지)
- 신고 3회 누적 시 자동 블라인드 처리
- 운영자 검토 후 복구 또는 영구 삭제
- 반복 위반 계정: 등록 기능 30일 정지

## 7. 보상 정책

> **단위**: 1포인트 = ₩1 (토스포인트, 즉시 지급)
> **지급 채널**: 앱인토스 비즈월렛 리워드 API. 자체 포인트 시스템(잔액/적립/출금) 없음.
> **사용자 노출**: Verdict 마이페이지에 누적 포인트 UI 미표시. 잔액/내역은 토스 앱에서 확인. 화면 카피는 "토스포인트"로 명시 (가짜 포인트 의심 차단).

### 7-1. 지급 기준

> **트랙 분리 원칙**: 일반 투표와 오늘의 투표는 보상 정책·지급 채널·운영 캠페인을 별도로 관리한다. 향후 Edge Function / API 엔드포인트도 트랙별로 분리해 프로모션·이벤트 시 한쪽만 가산 적용 가능하도록 설계한다.

#### 일반 투표 트랙

| 트리거 | 지급 포인트 | 캡 / 빈도 | 비고 |
|--------|------------|-----------|------|
| 일반 투표 참여 | 1P / 건 | 하루 3건까지 | 4건째부터는 보상 없음 (콘텐츠 다양성 유도) |
| 3건 모두 완료 보너스 | +1P | 1회 / 일 | 게이미피케이션 (총 4P / 일) |
| 연속 출석 3일 | +1P | 보너스 | 끊기면 0부터 |
| 연속 출석 7일 | +3P | 보너스 | 끊기면 0부터 |
| 연속 출석 30일 | +20P | 보너스 | 끊기면 0부터 |
| 일반 투표 등록 (1~2건) | 2P / 건 | 하루 2건까지 즉시 지급 | 1~2건째에만 보상 지급 |
| 일반 투표 등록 (3~10건) | **0P** | 광고 시청 후 등록 | 보상 미지급 — 진짜 궁금한 질문만 등록 유도 (§13-1) |
| 일반 투표 등록 (11건 이상) | — | **차단** | 일일 하드캡 — 도배 차단 |
| 등록한 일반 투표 100명 달성 보너스 | 100명당 +3P | 캡 30P / 건 | 양질 콘텐츠 인센티브 — 도배 시 무의미 |

#### 오늘의 투표 트랙

| 트리거 | 지급 포인트 | 캡 / 빈도 | 비고 |
|--------|------------|-----------|------|
| 오늘의 투표 후보 신청 (작성) | 5P / 건 | **1인 1일 1건** | 후보 풀에서 1건 선정되는 구조라 다수 등록 무의미 — 질 우선 |
| 오늘의 투표 당선 (선정) | +30P (작성 5P에 추가) + 참여 50명당 +5P | 캡 100P / 건 (작성 5P 별도) | 1명 / 일 — 운영자 최종 확정 후 지급 |

> 카테고리 다양성: 같은 카테고리만 3건 투표 시 보너스 미지급(소수 의견 가산점은 v1.2에서 도입 검토).
> 트랙별 트리거 ID는 `points_log.trigger`에서도 분리(`normal_*` vs `today_*`)해 향후 정산·통계 분리.

### 7-2. 어뷰징 방지 게이트 (토스포인트 직지급 전제)

토스포인트는 ₩1 = ₩1 즉시 비용 확정이라 환전 락업으로 어뷰저를 거를 수 없다. 따라서 **지급 게이트**에서 차단한다.

- **일일 합산 한도 — 이중화 (어뷰징 핀셋)**: 트리거별 자체 캡이 있는 보너스(100명 보너스, 오늘의 투표 당선)는 자연 한도가 있으니 별도 트랙으로 풀어주고, 행동 빈도 기반 트리거에만 좁은 한도를 건다.
  - **카테고리 1 — 행동 기반 한도: 30P / 일**
    - 적용 트리거: 일반 투표 참여 / 3건 완료 보너스 / 출석 스트릭(3·7·30일) / 일반 투표 등록 보상 / 오늘의 투표 후보 신청
    - 정상 사용자 천장 ~10P, 30일 스트릭 도달일에 한해 ~30P까지 닿음. 어뷰저는 명확히 컷오프.
  - **카테고리 2 — 콘텐츠 성과 기반 한도: 130P / 일**
    - 적용 트리거: 등록한 일반 투표 100명 달성 보너스(캡 30P/건) / 오늘의 투표 당선(캡 100P/건, 1명/일)
    - 발생 조건이 까다롭고 트리거별 자체 캡이 있어 어뷰징 위험 자연 차단. 정상 유저의 좋은 콘텐츠 보상이 컷되지 않도록 별도 한도.
  - **총 일일 천장: 160P** (정상 사용자 99%는 ~10P 수준)
- **디바이스 1:1 매핑**: 같은 디바이스에서 다계정 보상 수확 차단 (토스 콘솔 디바이스 핑거프린트 활용 가능 여부 확인 필요).
- **신규 가입자 보상 24h 지연**: 가입 후 24시간 동안은 보상 적립만, 지급은 보류.
- **투표 응답 유효성**: 단순 클릭 패턴(전부 A 선택 등) 탐지 시 보상 차감 또는 보류.
- **로그**: 사용자별 지급 사유 / 시각 / 금액 / 트리거 ID 전수 기록 (분쟁·정산 대응).

### 7-3. 단위 경제 (DAU 1,000 가정)

| 항목 | DAU 비용 | 일 비용 |
|------|---------|---------|
| 3투표 + 보너스 (4P) × 50% | ₩2 | ₩2,000 |
| 등록 (2P) × 10% | ₩0.2 | ₩200 |
| 오늘의 투표 당선 (평균 60P) × 1% | ₩0.6 | ₩600 |
| 출석 보너스 (평균 ₩1.5) | ₩1.5 | ₩1,500 |
| **포인트 지급 합계** | **₩4.3** | **₩4,300** |
| 광고 수익 (배너 + 리워드 + 전면 합산) | ₩50 | ₩50,000 |
| **마진** | | **₩45,700 (91%)** |

> 광고 도달률 100% 가정이라 실제 순마진은 50~60%로 보수적으로 추정.
> 월 지급 총액이 ₩100만 초과 시점부터 세무사 컨펌 필요(마케팅 비용 처리 / 부가세).

### 7-4. 운영 방침

- 초기 (0~1,000명): 포인트 재원 선투자 (예산 ₩100~200만 / 월).
- 성장기 (1,000명~): 광고 수익 일부를 포인트 재원으로 순환.
- 출시 전 확정: 토스 콘솔 보상 지급 약관 + 광고 SDK 약관 교차 확인 (특히 "광고 시청 → 토스포인트 직지급" 허용 여부).

## 10. 기술 스택 참고사항

- 플랫폼: 앱인토스 (Toss mini-app)
- AI 검열: OpenAI Chat Completions (`gpt-4o-mini`, JSON 모드 강제) — 1건당 약 ₩0.24 (sample 10개 기준)
- 인증: **토스 실명 인증(IdP) ↔ Supabase Auth(세션 관리자) 브릿지** — 1인 1계정 보장
- DB / 백엔드: Supabase (Postgres + Edge Functions + RLS)
- 광고: 앱인토스 비즈월렛 리워드 광고 연동
- 푸시 알림: 앱인토스 푸시 인프라 활용

### 인증 / 세션 아키텍처

```
사용자 (앱인토스 진입 → 토스 세션 자동 보유)
    ↓
클라이언트 → Edge Function에 토스 사용자 정보 전달
    ↓
Edge Function:
  1) Supabase Admin API로 auth.users upsert (toss_user_key 기준)
  2) public.users 동기화 (성별·연령대 동의분만 저장, 미동의 시 'undisclosed')
  3) Supabase JWT 발급
    ↓
클라이언트 → 받은 JWT로 Supabase 직접 호출 (RLS 자동 적용)
```

- 사용자에게 별도 Supabase 로그인 화면 노출 없음 — 토스 한 번 로그인이 전부
- 모든 RLS 정책은 `auth.uid()` 기반으로 본인 데이터만 접근 가능

## 12. 투표 보관 및 노출 정책

투표 타입별로 피드 노출 기간과 DB 보관 기간을 분리 운영한다.

### 12-1. 일반 투표

| 항목 | 정책 |
|------|------|
| 홈피드 노출 | **진행중 + 마감 후 24시간까지만** 노출, 이후 피드에서 자동 제거 |
| DB 보관 | 마감일로부터 **30일 후 자동 삭제** |
| 마이페이지 이력 | 최근 **30일치 이력**만 확인 가능 (내가 올린 투표 / 참여한 투표 모두) |
| 상세 페이지 접근 | DB 보관 기간 내에는 URL/마이페이지 진입 시 결과 확인 가능, 30일 경과 후에는 접근 불가 |

### 12-2. 오늘의 투표

| 항목 | 정책 |
|------|------|
| 홈피드 노출 | 발행일 진행중 카드 → 마감 후 **하단 "지난 오늘의 투표" 섹션**으로 이동 |
| DB 보관 | **마감일로부터 180일(약 6개월) 후 자동 삭제** (pg_cron 잡으로 운영) |
| 지난 오늘의 투표 | 홈피드 하단 섹션에서 날짜별 카드로 나열, 카드 탭 시 상세 결과 확인 (보관 기간 내) |
| 상세 페이지 접근 | 보관 기간(180일) 내에는 접근 가능 (재투표 불가, 결과만 노출), 이후 진입 불가 |
| 포인트 이력 | 투표 삭제 시 `points_log.related_vote_id`는 `set null`로 anonymize되어 정산 기록은 보존 |

### 12-3. 마감된 투표 미참여자 결과 공개

- 두 트랙 모두 **마감 후 미참여자**가 결과를 보려면 **광고 시청** 필요 (§4-2 참고)
- 일반 투표는 마감 후 24시간 이후 피드에서 사라지므로, 주된 진입 경로는 공유 링크 또는 마이페이지 이력
- 오늘의 투표는 "지난 오늘의 투표" 섹션을 통한 진입이 주된 경로

## 13. 광고 배치 정책

토스포인트 직지급 모델에서는 **리워드 광고 eCPM(₩6,000–15,000)** 이 전면(₩2,000–4,000)·배너(₩300–800)보다 압도적으로 높고, 사용자 통제권이 있어 이탈도 적다. 따라서 **리워드를 기본 포맷**으로 두고, 노출량 확보용으로 네이티브 배너·전면을 보조 사용한다.

### 13-1. 포맷별 사용처

| 위치 | 포맷 | 게이팅 / 빈도 | 근거 |
|------|------|---------------|------|
| 홈피드 카드 사이 (4~5번째마다) | **네이티브 배너** | 상시 | 토스 룩과 어울림. CTR이 일반 배너 대비 3~5배. eCPM ₩1,500–3,000. |
| 마감 투표 결과 보기 (미참여자) | **리워드** | 결과 1회당 1회 | 결과 노출의 정상 경로. 사용자가 능동 선택해 이탈·정책 이슈 모두 적음. (§4-2 참고) |
| 투표 등록 3~10건째 | **리워드** | 하루 3건째부터 10건째까지 매 건 (11건째부터 차단) | "한도 해제" 메커니즘은 리워드 정석. 광고 시청해도 포인트 미지급 — 진짜 궁금한 질문만 등록 유도. |
| 투표 직후 결과 화면 — 상세 통계 게이트 | **리워드** | 통계 펼치기 1회당 1회 | 성별·연령대별 통계를 리워드 게이트로 묶어 추가 광고 자리 확보. ROI 가장 높은 추가 자리. |
| 등록 완료 후 모달 | **전면 (skippable)** | 3건 등록당 1회 (frequency cap) | 전환 직후라 이탈 영향 적음. 캡 필수. |
| 앱 진입 1회 (스플래시 후) | **앱 오픈 (App Open)** | 하루 1회 (frequency cap) | 사용자 빈도 높을수록 누적 수익 ↑. 캡 없으면 앱 평점 직격. |

### 13-2. 금지 / 비권장 자리

- **투표 직후 자동 전면**: 결과 직전에 강제 전면을 끼우면 이탈 15~25%. 리워드 게이트 외에는 막지 않는다.
- **카드 탭마다 인터스티셜**: 핵심 행동을 가로막는 전면은 일체 금지.
- **마이페이지 진입 시 전면**: 토스 정책상 핵심 진입로 차단은 검수 리스크.

### 13-3. 광고 SDK 운영 원칙

- **AbortSignal 기반 취소**: 화면 언마운트·타임아웃·사용자 취소를 한 채널로 처리 (현재 `UnlockProvider`에서 구현됨).
- **로드 실패 / 타임아웃**: 사용자에게 토스트 안내 + 재시도 가능 ("광고 로드 시간이 초과됐어요").
- **임프레션 정합성**: 타임아웃·취소된 광고는 임프레션·보상 모두 미발생 보장 (어댑터 레이어에서 dispose).
- **frequency cap 저장소**: 클라이언트 로컬(localStorage) + 서버 이중 기록 (어뷰저는 로컬만으론 못 막음).

### 13-4. 광고/리워드 정책 교차 확인 (출시 전 필수)

토스 콘솔 보상 약관과 광고 SDK 약관을 교차 점검:

- "광고 시청 → 토스포인트 직지급" 흐름이 양쪽 모두 허용되는지.
- 일부 보상형 광고 SDK는 **자체 포인트로만 지급 가능**할 수 있음 — 이 경우 어댑터에서 자체 포인트 → 토스포인트 환전 단계 필요.
- 미성년자 / 신규 가입자 보상 제한 정책 (토스 측).
